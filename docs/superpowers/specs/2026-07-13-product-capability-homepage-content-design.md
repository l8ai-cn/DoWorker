# Agent Cloud 产品能力与首页内容设计

- **日期：** 2026-07-13
- **状态：** 已确认并进入实现；持续维护基线见 `docs/product/expert-platform-capability-map.md`
- **范围：** 产品定位、能力地图、首页信息架构、跨境电商、AI 教育、AI 伙伴、市场
- **维护原则：** 代码与已批准设计是事实来源；实现能力、可组装能力、规划能力必须分开展示

## 1. 产品判断
Agent Cloud 不是把多个 Agent 角色堆在一个界面里的聊天产品，也不是单纯的
Agent 启动器。它应被定义为：
> 一个以 Expert 为业务入口，把不同 Worker、模型、Skill、知识、系统连接和
> Workflow 组织成完整交付链路的 AI 工作平台。
用户面对的是一个能承担结果责任的 Expert，而不是先理解 Codex、Claude Code、
Skill、MCP、Runner 和 Pod。底层可以由一个或多个 Worker 协作，但产品的稳定
入口、配置复用、运行治理和市场分发单位应是 Expert。
首页核心文案建议：
> **把分散的 AI 能力，组织成真正完成工作的专家。**
>
> 在一个工作空间里连接模型、Worker、Skill、知识与业务系统，从目标拆解、
> 执行协作、人工确认到结果交付，打通完整工作链路。

## 2. 产品对象与职责
| 对象 | 用户价值 | 产品职责 |
| --- | --- | --- |
| Expert | 一个可反复调用的业务专家 | 固化目标、WorkerSpec、Skill、知识、权限、自动化等级和运行方式 |
| Worker | Expert 的执行实例 | 在隔离工作区内运行具体 Agent 类型并产生过程与结果 |
| Worker 类型 | 不同执行引擎 | 提供 Codex、Claude Code、Gemini CLI 等运行协议和配置 |
| Skill | 可组合的原子能力 | 定义完成某类任务的方法、工具约束和交付标准 |
| Knowledge | 组织上下文 | 向 Worker 挂载可治理、可复用的知识库 |
| MCP / 系统连接 | 外部动作能力 | 连接业务系统和工具，凭据保留在组织运行时 |
| Workflow | 可重复的业务流程 | 支持手动、API、定时触发，并控制并发、超时、沙箱和会话 |
| Goal Loop / Autopilot | 有边界的持续执行 | 使用验收条件、预算、迭代上限、无进展检测和人工接管闭环 |
| Market | 能力分发入口 | 让用户按业务结果发现并启用 Expert、Skill 和连接能力 |
核心链路：
```text
业务目标
  -> 选择或创建 Expert
  -> Expert 调用一个或多个 Worker
  -> Worker 装载模型 + Skill + Knowledge + MCP
  -> Workflow 编排触发、上下文、并发与持续运行
  -> 关键节点人工确认
  -> 交付文档、代码、图片、音频、视频、数据或系统动作
  -> 保存证据并复用为下一次运行
```

## 3. 当前支持的 Worker 类型
正式目录当前包含 12 类 Worker：

- 代码与复杂执行：Codex CLI、Claude Code、Gemini CLI、Aider、Cursor CLI。
- 通用与开放执行：DoAgent、OpenCode、Grok Build、Hermes、OpenClaw。
- 循环与特定模型协议：Loopal、MiniMax CLI。

它们是执行引擎，不直接等同于“编剧”“运营”“教师”等业务角色。业务角色由
Expert 对 Worker、Skill、知识、工具和流程封装后形成。

## 4. 能力边界
### 已实现的平台原生能力
- 多种 Worker 类型的统一创建、运行和会话接入。
- 自托管 Runner、隔离工作区、PTY / ACP、模型绑定和凭据引用。
- Expert 的创建、发布、复用和运行，支持 Skill、知识、环境与自动化等级。
- Skill 目录、版本化包、Agent 兼容约束和运行时精确装载。
- Knowledge Base 挂载、MCP 配置、仓库和环境能力组合。
- Workflow 的手动、API、Cron 触发，以及并发、超时、持久沙箱和会话策略。
- Autopilot 的暂停、恢复、人工接管、审批等待和熔断状态。
- Goal Loop 的验收条件、验证命令、迭代/Token/时间预算和无进展检测。
- Ticket、Channel、Mesh 和富文档 Block Store，用于任务、协作、证据与交付物。
- 图片、文件、视频、音频、嵌入、表格、图表等内容的存储与呈现。
- 当前专家应用市场及 Skill 目录。

### 可通过 Expert + Skill + MCP 组装的业务能力
| 能力族 | 可形成的 Expert | 当前表达边界 |
| --- | --- | --- |
| 编程与研发 | 开发、测试、代码评审、发布、双仓同步 | 当前证据最完整，可作为首要可信案例 |
| 文档与办公 | 研究报告、方案、合同初审、表格、演示文稿 | 需要对应文档 Skill/工具；平台负责流程、知识和交付 |
| 内容与创意 | 选题、写作、脚本、分镜、编导协作 | Expert 负责链路；生成质量取决于接入模型和 Skill |
| 图片、音频、视频 | 图片生成、配音、字幕、视频片段与审核 | 平台可承载媒体结果，但不应声称内置媒体生成模型或编辑器 |
| 数据与运营 | 数据查询、日报、巡检、异常处理 | 需要数据库、表格或业务系统连接与明确权限 |
| 研究与知识 | 搜索、资料整理、知识问答、知识维护 | 可结合 Knowledge、检索、文档和外部搜索工具 |
| 教育与培训 | 课程研究、大纲、课件、实验、练习和评审 | 适合作为完整 Expert 应用链路，而不是单个“教师 Agent” |
| AI 伙伴 | 持续目标、跨部门协作、跨系统操作和进展同步 | 以共享上下文、能力组合、人工确认和可追溯证据为核心 |

### 已设计但尚未实现
多租户行业市场、品牌市场、Space 专区、四类 Listing、Entitlement、Quota
Ledger、两阶段安装和市场 Runtime Bridge 已完成产品与架构设计，但实现尚未
开始。首页可以表达发展方向，不能将其写成已经上线的功能。

## 5. 四个菜单的产品规划
### 跨境电商
目标不是放置选品、文案、客服等多个割裂角色，而是提供“跨境增长 Expert”，
串联市场研究、商品准备、内容生产、上架协同、投放素材、客服知识和经营复盘。
典型链路：
```text
市场/竞品研究 -> 选品判断 -> 多语言商品内容 -> 图片/视频素材
-> 人工确认品牌与合规 -> 上架/投放系统连接 -> 客服知识 -> 经营日报
```
首批功能应包括：多市场研究、Listing 本地化、素材任务包、合规检查点、客服
知识库、日报 Workflow。ERP、广告平台和店铺写入必须在对应连接真实存在后展示。

### AI 教育
定位为“从教学目标到可交付课程”的课程研发工作台，而不是一个通用问答教师。
典型链路：
```text
受众与目标 -> 课程研究 -> 大纲与任务 -> 课件/文档
-> 实验与练习 -> 图片/音频/视频资源 -> 教师审核 -> 发布与持续更新
```
首批功能应包括：课程架构 Expert、课件 Expert、实验 Expert、练习评测 Expert、
多媒体素材协作、课程知识库和版本化交付。人工确认放在大纲、事实准确性、评价
标准和正式发布四个节点。

### AI 伙伴
定位为围绕持续业务目标与团队协作的 Expert。它共享组织上下文，组合研究、
文档、数据、内容和系统操作等能力，在关键节点主动同步并请求确认。用户管理的
是共同目标、协作边界和结果，不是一个被模拟或替代的岗位。
典型链路：
```text
共同设定目标 -> 读取组织知识 -> 执行跨工具任务 -> 主动同步进展
-> 人工确认高风险动作 -> 写回系统 -> 交付结果与可查验证据
```
首批功能应包括：研究助理、内容运营、文档办公、数据分析、研发交付、运维巡检。
Workflow 提供重复执行，Goal Loop 提供有上限的持续推进，Autopilot 提供人工
接管和审批状态。

### 市场
当前市场应聚焦“按结果发现并启用专家应用”，首页可真实展示软件交付、多 Worker
协作、双仓同步三个现有应用，并补充 Skill 浏览入口。
市场的目标结构是：应用、Skill、系统连接、资源。详情页先展示结果、示例、
权限、所需账号和验证状态，再展示底层 Worker、Skill 和 MCP 装配信息。跨境
电商、AI 教育和 AI 伙伴未来都应作为市场专区，而不是与市场割裂的模板页面。

## 6. 首页信息架构与文案
1. **Hero：统一入口。** 标题使用“把分散的 AI 能力，组织成真正完成工作的专家”。
   主视觉展示一个业务目标进入 Expert 工作台后，贯通 Skill、知识、系统、人工
   确认和结果交付，不再以多个角色头像作为主体。
2. **产品公式。** `Expert = Worker + Model + Skills + Knowledge + Tools + Workflow`，
   解释 Expert 是业务入口，Worker 是运行实例。
3. **四个菜单入口。** 跨境电商、AI 教育、AI 伙伴、市场，各自展示一条完整链路
   和明确交付物。
4. **能力光谱。** 编程、文档办公、研究知识、内容创作、图片、音频、视频、数据
   运营；使用“可组装的专家能力”而非“全部原生内置”。
5. **一处贯通完整链路。** 目标、执行、协作、检查点、交付和复用都在同一空间。
6. **Expert 系统。** 展示模型、Worker 类型、Skill、知识、系统连接、权限和流程
   如何被固化为可复用 Expert。
7. **Workflow 与持续执行。** 手动/API/定时触发，支持持久上下文、并发策略、
   预算、停止条件、人工接管和失败证据。
8. **人工确认与结果交付。** 高风险动作需要确认；结果包含过程、证据、文件与
   状态，不以“Agent 已完成”作为交付标准。
9. **自托管与治理。** Runner 部署在受控基础设施，密钥使用引用，执行和数据面
   分离，组织保留运行与数据控制权。
10. **多 Worker 运行底座。** 展示 12 类 Worker 兼容性，但放在业务价值之后。
11. **市场。** 从已上线专家应用切入，说明能力可以安装、复用和组织内分发。
12. **最终行动。** “创建你的第一个 Expert”与“浏览专家市场”；首页不展示定价。

## 7. 视觉与交互方向
- 参考 Sierra 的结果导向、沉浸式产品叙事和高密度操作感，不复制其品牌语言。
- 采用已确认的 C 方向：深色技术控制面，但 Hero 首先讲业务目标与 Expert。
- 使用石墨黑、近白、薄荷绿和少量琥珀色，分别表达工作面、信息、执行与确认。
- 视觉主体是一条正在运行的工作链路，不是装饰性网格、渐变球或角色卡片墙。
- 首页营销路由保持零 WASM；交互使用确定性演示数据，并适配移动端与减弱动画。

## 9. 持续维护规则
- Worker 目录、Expert 字段、Skill/MCP/Knowledge 装载、Workflow 或市场状态变化时更新。
- 每次首页文案修改先标注事实等级：`已实现`、`需安装能力`、`规划中`。
- 新业务能力必须记录所需 Worker、Skill、知识、系统连接、人工检查点和交付物。
- 不使用“全自动”“无需人工”“支持所有系统”等无法由代码和验证证据支撑的表述。
