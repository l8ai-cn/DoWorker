# Loop 积木化编程需求规格说明书

- 日期：2026-07-13
- 更新：2026-07-15
- 状态：V1 已实现，通用组件与 AI 生成进入后续阶段
- 产品名：Loop 工作台
- 执行对象：AgentsMesh Goal Loop

## 1. 产品目标

Loop 工作台让用户通过积木或 LoopScript 声明智能体执行过程，并保持两种编辑方式
双向一致。Loop 描述目标、任务、验证、预算和失败策略；Worker 是运行环境，只在
启动时选择，不属于 LoopScript、AST、积木连接或发布版本。

系统必须支持以下闭环：

1. 用户通过积木创建或修改 Loop。
2. 系统生成规范 LoopScript，并可从合法代码重建积木。
3. 用户可编辑积木内的短参数和代码中的完整参数。
4. 系统在编译时返回规范源码、结构化 AST 和定位诊断。
5. 用户启动时选择运行环境，后端校验后创建并执行 GoalLoop。
6. 验证器决定是否完成，失败证据驱动下一轮，不接受 AI 自报成功。

## 2. 用户与核心场景

主要用户是需要重复委派 AI 工作的团队成员。用户只需回答：

- 要完成什么？
- 哪些证据代表完成？
- 最多执行多久、多少轮、多少 Token？
- 失败后暂停人工处理还是直接失败？
- 本次启动使用哪个运行环境？

运行环境的选择不得改变 Loop 的编排语义或代码内容。

## 3. V1 验收场景

### 场景 A：积木生成代码

Given 用户在工作台放置完整的循环、任务、验证和边界积木
When 用户修改积木参数
Then 系统生成无 Worker 声明的规范 LoopScript，并保留节点标识

### 场景 B：代码重建积木

Given 用户输入合法 LoopScript
When 后端编译成功
Then 工作台按 AST 重建积木，保持执行顺序、参数和节点标识

### 场景 C：阻止非法代码覆盖

Given 当前积木对应最后一次合法程序
When 用户输入非法语法
Then 系统展示中文诊断，保留最后一次合法积木，不静默覆盖或降级

### 场景 D：运行环境独立绑定

Given 当前 Loop 已通过编译
When 用户点击“运行循环”
Then 系统弹出运行环境选择框，并仅在启动请求中提交快照标识

### 场景 E：运行环境失效

Given 用户选择的运行环境已停用、过期或不属于当前组织
When 用户确认启动
Then 后端拒绝创建 GoalLoop，并返回明确错误

### 场景 F：确定性循环

Given 验证未通过且预算仍可用
When 控制器进入下一轮
Then 系统将失败证据发送给同一运行环境，并通过稳定命令标识保证重放幂等

## 4. V1 积木目录

| 分类 | 积木 | 参数 | LoopScript |
| --- | --- | --- | --- |
| 程序 | 循环 | 名称 | `loop <id> {}` |
| 控制 | 重复直到 | 名称、最大次数、条件 | `repeat` |
| 任务 | 智能体任务 | 名称、提示词 | `agent` |
| 验证 | 命令验证 | 名称、命令、验收说明 | `verify` |
| 边界 | 执行边界 | 轮数、Token、时长、无进展、同错 | `limits` |
| 失败 | 失败处理 | 暂停或失败 | `on_failure` |

约束：

- 工具箱和 AST 不包含 Worker 积木。
- 一个程序只允许一个顶层 Loop。
- Loop 必须包含边界、循环体、验证器和失败策略。
- 重复结构必须有局部最大次数，并同时受全局边界约束。
- 所有节点使用稳定 `node_id`；复制产生的重复标识必须阻止编译。
- 未知积木、孤立积木、非法引用和无退出条件循环都是阻塞错误。

## 5. LoopScript 边界

LoopScript 是受限声明式语言，不允许任意函数、动态加载、反射或直接网络访问。
外部行为必须通过平台登记的能力节点表达。

V1 明确禁止：

- `worker <alias> = snapshot(<id>)`
- `agent <id>(using: <worker>)`
- Secret 字面量
- 无边界循环
- 客户端直接提交可信执行计划

后端编译器是规范源码、AST 和诊断的权威来源。

## 6. 运行与 AI 执行

启动请求由两部分组成：

- `source`：不含运行环境的 LoopScript。
- `worker_spec_snapshot_id`：本次运行使用的环境快照。

后端必须在创建 GoalLoop 前校验快照的组织归属、可用性和新鲜度。Web 端以十进制
字符串保存快照标识，编码 protobuf 时再转换为 `int64`，禁止经过 JavaScript
`number`。

控制器必须依赖验证结果、预算和持久化状态推进。命令重试、Runner 重连和服务重启
不得重复执行已确认的副作用。

## 7. 中文与多语言

V1 默认呈现中文：

- 工具箱分类、积木字段、按钮、状态和诊断摘要使用中文。
- LoopScript 关键字、标识符、命令和用户提示词保持原值。
- 协议状态不得直接显示，必须映射为用户语言。

通用组件阶段必须将显示文案从积木定义和 React 组件中抽离为语言资源，并允许宿主
注入语言包。切换语言只能改变显示，不得改变 AST、源码或节点标识。

## 8. AI 生成

AI 生成阶段必须输出结构化 Loop AST 或受约束的语义修改命令，不得直接生成
Blockly 工作区 JSON。生成流程必须经过后端编译和验证后才能进入工作台。

最低能力：

- 根据自然语言目标生成 Loop 草稿。
- 根据诊断修复指定节点或参数。
- 解释当前循环、预算和验证路径。
- 展示变更预览，由用户确认后应用。

AI 不得修改验证器以规避失败，不得删除预算边界，也不得自行选择运行环境。

## 9. 通用组件

通用组件阶段应将工作台拆为可复用组件，宿主负责提供：

- 编译、运行环境列表和启动接口。
- 当前组织、权限和语言资源。
- AI 生成或修改接口。
- 自定义积木目录和发布策略。

组件负责编辑器切换、积木投影、代码编辑、诊断定位、运行环境选择和变更预览。
业务状态仍由 Rust Core 和后端领域服务持有，不在 React 本地建立第二套权威状态。

## 10. 安全与停止条件

- AI 完成声明不构成成功证据。
- 验证器退出码或受信结构化结果才可确认成功。
- Controller 强制执行轮数、Token、时间、无进展和同错上限。
- 已发布程序和自定义积木不可原地修改。
- Secret 只能使用引用，不进入源码、AST、日志或 AI 提示。
- 不提供 fallback、静默跳过或“尽力执行”模式。

## 11. 交付阶段

- V1：积木与代码双向联动、中文界面、运行环境解耦、真实 GoalLoop 执行。
- V1.1：通用组件接口和语言资源注入。
- V1.2：AI 生成、修改预览和确认应用。
- V2：版本化自定义积木、Skill/MCP/平台能力节点和发布治理。
