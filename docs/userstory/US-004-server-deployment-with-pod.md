## 1. 概述

### 1.1 用户故事

**作为** 一名 DevOps 工程师或开发者，
**我希望** 通过 Do Worker Pod 执行服务器部署任务，
**以便** 让 AI Agent 自动化处理部署流程，减少人工操作，降低部署风险。

### 1.2 价值主张

- **自动化部署**: AI Agent 理解部署意图，自动执行复杂操作
- **智能故障处理**: 遇到问题时自动诊断并尝试修复
- **实时可观测**: 全程监控部署进度和服务器状态
- **安全可控**: 敏感操作需确认，支持回滚机制

## 2. 部署场景

### 2.1 支持的部署类型

| 类型 | 说明 | 典型工具 |
|------|------|----------|
| **应用部署** | 部署应用代码到服务器 | SSH, rsync, Docker |
| **容器部署** | 部署到 Kubernetes/Docker 环境 | kubectl, docker-compose |
| **基础设施** | 配置服务器、网络、存储 | Ansible, Terraform |
| **数据库变更** | 执行数据库迁移脚本 | migrate, flyway |
| **配置更新** | 更新配置文件、环境变量 | SSH, ConfigMap |

## 3. 前置条件

| 条件 | 说明 |
|------|------|
| Runner 已部署 | Runner 需要有访问目标服务器的网络权限 |
| 凭证已配置 | SSH 密钥、kubeconfig、云服务凭证等已配置 |
| 部署脚本就绪 | 项目中存在部署相关脚本或配置文件 |
| 权限已授予 | Runner 运行用户有执行部署操作的权限 |

### 4.2 详细步骤

#### Step 1: 选择部署环境

1. 进入项目部署页面
2. 选择目标环境：
   - **Development**: 开发环境
   - **Staging**: 预发布环境
   - **Production**: 生产环境（需额外确认）
3. 查看环境配置和当前状态

**环境配置示例:**
```yaml
environments:
  development:
    servers:
      - host: dev.example.com
        user: deploy
        port: 22
    deploy_path: /var/www/app
    branch: develop

  staging:
    servers:
      - host: staging.example.com
        user: deploy
        port: 22
    deploy_path: /var/www/app
    branch: main

  production:
    servers:
      - host: prod-1.example.com
        user: deploy
        port: 22
      - host: prod-2.example.com
        user: deploy
        port: 22
    deploy_path: /var/www/app
    branch: main
    requires_approval: true
```

#### Step 2: 配置部署参数

1. 选择部署版本（分支/Tag/Commit）
2. 配置部署选项：
   - 是否执行数据库迁移
   - 是否清理缓存
   - 是否重启服务
   - 滚动部署 vs 全量部署
3. 设置部署超时时间

**部署配置 UI:**
```

│ 部署配置                                                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │

│ 部署版本:     [main ▼]  commit: a1b2c3d                         │
│                                                                 │
│ ┌─────────────────────────────────────────────────────────────┐ │
│ │ 部署选项                                                     │ │
│ │ ☑ 执行数据库迁移                                             │ │
│ │ ☑ 清理应用缓存                                               │ │
│ │ ☑ 重启服务                                                   │ │
│ │ ☐ 滚动部署 (逐台更新)                                        │ │
│ └─────────────────────────────────────────────────────────────┘ │
│                                                                 │

│                                                                 │
│                              [取消]  [开始部署]                  │
└─────────────────────────────────────────────────────────────────┘
```

#### Step 3: 启动部署 Pod

1. 系统创建部署专用 Pod
2. 加载部署凭证和配置
3. Agent 准备部署环境

**Pod 配置:**
```yaml
pod:
  type: deployment
  agent: claude-code

  sandbox:
    type: tempdir  # 部署任务使用临时目录

  secrets:
    - name: SSH_PRIVATE_KEY
      from: vault/deploy/ssh-key
    - name: KUBECONFIG
      from: vault/deploy/kubeconfig

  env:
    DEPLOY_ENV: staging
    DEPLOY_VERSION: a1b2c3d

  capabilities:
    - ssh
    - kubectl
    - docker
```

#### Step 4: Agent 执行部署

1. Agent 分析部署任务
2. 生成部署计划并展示
3. 用户确认后执行

**Agent 部署计划:**

## 7. 回滚机制

### 7.1 自动回滚触发条件

| 条件 | 触发动作 |
|------|----------|
| 健康检查连续失败 3 次 | 自动回滚到上一版本 |
| 部署超时 | 提示用户选择回滚或继续 |
| 错误率突增 (>5%) | 告警并建议回滚 |
| 用户手动触发 | 立即执行回滚 |

### 7.2 回滚执行流程

```
[回滚执行]

触发原因: 健康检查失败 - /health 返回 503

回滚目标: v1.2.2 (上一个稳定版本)

执行步骤:
1. ✓ 停止当前部署
2. ✓ 恢复代码到 v1.2.2
3. ✓ 回滚数据库迁移 (如适用)
4. ✓ 重启服务
5. ✓ 健康检查通过

回滚完成，服务已恢复到 v1.2.2
当前状态: 正常运行

建议: 请检查 v1.2.3 的变更，修复问题后重新部署
失败日志已保存: /var/log/deploy/2026-01-16-failed.log
```

## 9. 验收标准

| # | 标准 | 验证方式 |
|---|------|----------|
| 1 | 支持 SSH 连接到目标服务器 | 连接测试 |
| 2 | 支持 kubectl 操作 K8s 集群 | 集群操作测试 |
| 3 | 部署日志实时展示，延迟 < 1 秒 | 性能测试 |
| 4 | 生产环境部署需要额外确认 | 权限测试 |
| 5 | 健康检查失败自动触发回滚 | 故障注入测试 |
| 6 | 部署历史可追溯 | 审计日志验证 |
| 7 | 凭证安全存储，不明文展示 | 安全审计 |

## 10. 异常处理

| 场景 | 处理方式 |
|------|----------|
| SSH 连接失败 | 检查网络和凭证，提示用户排查 |
| 磁盘空间不足 | 部署前检查，空间不足时阻止部署 |
| 构建失败 | 显示错误日志，终止部署 |
| 数据库迁移失败 | 自动回滚迁移，通知用户 |
| 服务启动失败 | 收集日志，触发回滚流程 |
| 健康检查超时 | 重试 3 次后触发回滚 |
| 网络中断 | 保存状态，支持断点续传 |
