## 1. 概述

### 1.1 用户故事

**作为** 一名 QA 工程师或开发者，
**我希望** 使用 Agent Cloud Pod 运行端到端（E2E）测试，
**以便** 在隔离环境中验证功能完整性，支持单 Pod 快速验证或多 Pod 并行加速。

### 1.2 价值主张

- 隔离环境确保测试结果可重复
- 多 Pod 并行执行缩短测试时间
- 实时查看测试进度和日志输出
- 测试结果自动汇总和报告

---

## 2. 前置条件

| 条件 | 说明 |
|------|------|
| 用户已登录 | 拥有有效的 Agent Cloud 账户 |
| Runner 已注册 | 至少有一个在线的 Runner（多 Pod 模式需要足够资源） |
| 测试用例已编写 | 项目中存在 E2E 测试代码 |
| 测试配置就绪 | E2E 测试相关依赖和配置已准备 |

---

## 3. 使用场景

### 场景 A: 单 Pod 运行 E2E 测试

适用于快速验证、调试单个测试用例或资源有限的情况。

### 场景 B: 多 Pod 并行运行 E2E 测试

适用于完整测试套件执行、CI/CD 流水线或需要缩短测试时间的场景。

---

## 4. 场景 A: 单 Pod E2E 测试

### 4.1 流程图

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ 选择测试用例 │ ──► │  启动 Pod   │ ──► │  执行测试   │ ──► │  查看结果   │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
```

### 4.2 详细步骤

#### Step 1: 选择测试用例

1. 进入项目 E2E 测试页面
2. 浏览测试用例列表
3. 选择要运行的测试：
   - 全部测试
   - 按标签筛选（smoke、regression、feature）
   - 按文件/目录选择
   - 单个测试用例

**示例配置:**
```yaml
test_selection:
  type: specific      # all | tagged | specific
  targets:
    - e2e/account/login.spec.ts
    - e2e/account/register.spec.ts
  tags: [smoke]
```

#### Step 2: 配置并启动 Pod

1. 选择 Agent 类型（推荐 Claude Code 执行测试命令）
2. 配置测试环境：
   - 目标分支
   - 环境变量（测试数据库、API 端点等）
   - 沙箱类型
3. 点击「启动测试」

**Pod 配置示例:**
```yaml
pod:
  agent: claude-code
  sandbox:
    type: worktree
    branch: feature/user-auth
  env:
    TEST_BASE_URL: http://localhost:3000
    TEST_DB_URL: postgres://test:test@localhost:5432/test
  command: |
    pnpm install
    pnpm test:e2e --reporter=html
```

#### Step 3: 执行测试并实时监控

1. Pod 启动后进入测试执行阶段
2. 实时显示：
   - 终端输出（测试日志）
   - 测试进度条
   - 当前执行的测试用例
3. 用户可随时：
   - 查看详细日志
   - 中断测试执行
   - 发送调试命令

**执行流程:**
```
Pod 启动
    │
    ▼
安装依赖 (pnpm install)
    │
    ▼
执行测试 (pnpm test:e2e)
    │
    ├── Test 1: Login ✓ (2.3s)
    ├── Test 2: Register ✓ (1.8s)
    ├── Test 3: Logout ✓ (0.9s)
    └── Test 4: Password Reset ✗ (3.2s)
    │
    ▼
生成报告
    │
    ▼
Pod 终止
```

#### Step 4: 查看测试结果

1. 测试完成后显示汇总：
   - 通过/失败/跳过数量
   - 总执行时间
   - 失败用例详情
2. 下载测试报告（HTML/JSON）
3. 查看失败用例的截图/视频（如有）

## 5. 场景 B: 多 Pod 并行 E2E 测试

### 5.1 流程图

```
                    ┌─────────────┐
                    │ 测试分片策略 │
                    └──────┬──────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
    │   Pod 1     │ │   Pod 2     │ │   Pod 3     │
    │ Shard 1/3   │ │ Shard 2/3   │ │ Shard 3/3   │
    └──────┬──────┘ └──────┬──────┘ └──────┬──────┘
           │               │               │
           └───────────────┼───────────────┘
                           ▼
                    ┌─────────────┐
                    │  结果汇总   │
                    └─────────────┘
```

### 5.2 详细步骤

#### Step 1: 配置分片策略

1. 选择并行度（Pod 数量）
2. 配置分片策略：
   - **按文件分片**: 测试文件平均分配到各 Pod
   - **按时间分片**: 根据历史执行时间智能分配
   - **按标签分片**: 不同标签的测试分配到不同 Pod

**分片配置示例:**
```yaml
parallel:
  shards: 3                    # Pod 数量
  strategy: duration_based     # file_based | duration_based | tag_based
  balance: true                # 负载均衡

  # 按标签分片示例
  tag_shards:
    - name: fast
      tags: [smoke, unit]
      pods: 1
    - name: slow
      tags: [integration, e2e]
      pods: 2
```

#### Step 2: 启动多 Pod

1. 系统根据分片策略创建多个 Pod 请求
2. 调度器分配 Runner（可能分配到不同 Runner）
3. 各 Pod 并行启动和初始化

**调度流程:**
```
Backend                     Runner A              Runner B
   │                           │                     │
   │── create_pod (shard 1) ───►│                     │
   │── create_pod (shard 2) ───►│                     │
   │── create_pod (shard 3) ────────────────────────►│
   │                           │                     │
   │◄── pod_created ───────────│                     │
   │◄── pod_created ───────────│                     │
   │◄── pod_created ─────────────────────────────────│
```

#### Step 3: 并行执行与监控

1. 统一监控面板显示所有 Pod 状态
2. 各 Pod 独立执行分配的测试分片
3. 实时更新各 Pod 进度

**监控界面示例:**
```
┌─────────────────────────────────────────────────────────────┐
│ E2E 测试执行中 - 3 个 Pod 并行                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ Pod 1 (Shard 1/3)  ████████████░░░░░░░░  60%  12/20 passed │
│ Pod 2 (Shard 2/3)  ██████████████████░░  90%  18/20 passed │
│ Pod 3 (Shard 3/3)  ████████████████░░░░  80%  16/20 passed │
│                                                             │

│ 预计剩余时间: 2 分钟                                         │
└─────────────────────────────────────────────────────────────┘
```

#### Step 4: 结果汇总与报告

1. 所有 Pod 完成后自动汇总
2. 合并各分片的测试结果
3. 生成统一的测试报告

**汇总报告:**
```
┌─────────────────────────────────────────────────────────────┐
│ E2E 测试报告                                                │
├─────────────────────────────────────────────────────────────┤
```

## 7. 验收标准

### 7.1 单 Pod 模式

| # | 标准 | 验证方式 |
|---|------|----------|
| 1 | 可选择运行全部或部分测试用例 | UI 功能测试 |
| 2 | 测试日志实时展示，延迟 < 500ms | 性能测试 |
| 3 | 测试完成后生成可下载的报告 | 功能验证 |
| 4 | 支持中途取消测试执行 | 功能验证 |

### 7.2 多 Pod 模式

| # | 标准 | 验证方式 |
|---|------|----------|
| 1 | 支持 2-10 个 Pod 并行执行 | 功能测试 |
| 2 | 分片策略正确分配测试用例 | 日志验证 |
| 3 | 各 Pod 进度独立显示 | UI 验证 |
| 4 | 结果汇总准确，无遗漏 | 数据验证 |
| 5 | 单 Pod 失败不影响其他 Pod | 故障注入测试 |

---

## 8. 异常处理

| 场景 | 单 Pod | 多 Pod |
|------|--------|--------|
| Pod 启动失败 | 提示错误，支持重试 | 标记失败，其他 Pod 继续 |
| 测试超时 | 终止并记录超时用例 | 单 Pod 超时不影响整体 |
| Runner 断连 | Pod 状态变为 unknown | 重新分配到其他 Runner |
| 部分 Pod 失败 | N/A | 汇总时标记失败分片 |

---

## 9. 性能对比

### 假设: 60 个测试用例，单用例平均 15 秒

| 模式 | Pod 数量 | 预计时间 | 效率 |
|------|----------|----------|------|
| 串行 | 1 | 15 分钟 | 基准 |
| 并行 | 2 | 7.5 分钟 | 2x |
| 并行 | 3 | 5 分钟 | 3x |
| 并行 | 6 | 2.5 分钟 | 6x |

> 实际效率受 Pod 启动时间、测试分布不均等因素影响
