# 通用 Agent 助手 —— 功能补齐与专家库规划

## 一、当前定位反思

现有 Demo 偏「Coding Agent 控制台」（项目 = Git 仓库，任务 = 代码变更）。若要做**通用办公 Agent 助手**，需要把「项目」抽象为「工作空间/上下文」，把「专家 Agent」作为一等公民。

## 二、缺失功能盘点（按优先级）

### P0 · 通用化必备
1. **专家库（Expert Library）** —— 本次重点，见第三节
2. **工作空间类型泛化** —— 项目除了 Git 仓库，还应支持：
   - 文档工作区（Notion / Google Drive / 本地文件夹）
   - 数据工作区（表格 / 数据库连接）
   - 通讯工作区（邮箱 / 日历 / IM）
   - 浏览器工作区（网站访问 + 自动化）
3. **知识库 / 记忆（Memory）** —— 长期记忆 + 项目级记忆 + 用户偏好
4. **文件与产物（Artifacts）** —— 附件上传、Agent 产出的文档/图片/表格集中管理
5. **凭证与连接器（Connectors）** —— Gmail / Notion / Slack / Jira / GDrive 授权

### P1 · 交互与协作
6. **Ask User 表单**（上一轮已讨论）—— 动态表单/审批
7. **定时任务 & 触发器** —— Cron / Webhook / 邮件到达 / 文件变更触发
8. **多 Agent 协作** —— 主 Agent 调度子 Agent（Manager-Worker）
9. **审批中心** —— 所有待审操作聚合（工具调用、发信、付款、写文件）
10. **通知中心** —— Push / 邮件 / IM 摘要，支持"完成即通知"

### P2 · 可观测与治理
11. **成本 & Token 用量** —— 按项目/Agent/任务统计
12. **执行回放 & 分享** —— 一次任务生成可分享链接
13. **模板市场** —— 常用 Prompt / Workflow 模板
14. **数据导出** —— 会话/产物导出 md/pdf
15. **权限与团队** —— 多用户、角色、访问控制

### P3 · 增强体验
16. **快捷指令（Shortcuts）** —— 类似 iOS 快捷指令，链式调用专家
17. **对话中打断/追问** —— streaming 中随时插话
18. **多语言 & 语音输出** —— TTS 播报结果
19. **离线草稿** —— 无信号时排队，联网自动派发

### 3.4 数据模型（mock 层）

```ts
interface Expert {
  id: string;
  name: string;
  avatar: string;         // emoji 或图片
  category: "office" | "research" | "dev" | "life" | "creative" | "data";
  tagline: string;        // 一句话简介
  description: string;    // markdown
  author: string;
  rating: number;
  usageCount: number;
  capabilities: string[]; // 能力标签
  connectors: Array<{     // 依赖的连接器
    id: string;
    name: string;
    required: boolean;
    connected: boolean;
  }>;
  systemPrompt?: string;  // 内部
  tools: string[];        // 可用工具白名单
  suggestedPrompts: string[];
  recentRuns?: number;
  featured?: boolean;
  official?: boolean;
}
```

### 3.5 与现有流程的衔接

- **从专家库发起任务** → 跳转到 `/new?expert=<id>&project=<可选>`
- `/new` 页顶部 pill 由「选择项目」拓展为「专家 + 项目」二段选择：
  - 专家决定 *谁* 来做（能力/工具/prompt）
  - 项目决定 *在哪个上下文* 做（工作区/文件/凭证）
- **首页新增** 「专家库」入口 + 「最近使用的专家」横向滚动列表
- **任务详情页** 顶部显示所用专家 + 项目双 chip

## 四、建议的实施顺序

| 阶段 | 内容 | 价值 |
|---|---|---|
| **Step 1** | 专家库列表 + 详情 + mock 数据（8-10 个专家）| 立刻能演示"通用助手"定位 |
| **Step 2** | `/new` 支持 `?expert=` 预选，UI 加专家 chip | 打通"选专家→派任务"闭环 |
| **Step 3** | 首页加"最近使用的专家"row + 入口卡片 | 提升发现性 |
| **Step 4** | Ask User 表单事件类型（上一轮规划）| 体现动态交互 |
| **Step 5** | 连接器（Connectors）页面（mock 授权状态）| 支撑非 coding 能力 |
| **Step 6** | 审批中心聚合页 | 治理完整性 |

## 五、需要你确认的点

1. **专家库是独立 Tab，还是首页 section？** 建议独立 Tab（因为是核心入口）。
2. **项目 vs 专家的关系**：一个任务 = 一个专家 + 可选项目？还是一个专家可绑定默认项目？
3. **是否允许用户自定义专家**（后续 P1）？现在先只做官方专家库？
4. **首批要 mock 哪些专家**？我建议：邮件助手、会议纪要、深度研究员、数据分析、PPT 制作、合同审查、Codex（沿用）、旅行规划师、招聘助手、日报生成 —— 10 个覆盖办公/研究/开发/生活。

确认后我按 Step 1 + Step 2 先落地专家库页面和任务派发衔接。
